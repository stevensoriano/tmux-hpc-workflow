Quick reference for the tmux workflow on hpc-login. Project sessions: proj_a proj_b proj_c sandbox proj_d proj_e — each has 4 windows: shell (two side-by-side panes), agent (two side-by-side panes for parallel agents; proj_d has three), queue (squeue dashboard on top, sacct recent history below), logs.
Admin session: tadmin — different layout for cluster management: ops, agent (three side-by-side panes), mon, edit, cluster. See Admin.

The prefix key core concept

Every binding starts with the prefix: F12.

Tap it (release), then tap the next key. Don't hold F12 down.

Status bar flashes when you've pressed it. If you're ever stuck inside a menu or popup, Esc or q always exits.

Shorthand: F12 X means "tap F12, release, tap X."

Two no-prefix shortcuts. Faster than the prefix; use these constantly.
  • Shift+→ / Shift+← — cycle to next / previous window
  • Alt+↑ ↓ ← → — move between panes

The session tab strip (top status row). Your status bar now has two rows. The top row is a persistent strip of all sessions — proj_a │ proj_c │ … │ tadmin │ … — with the one this client is attached to highlighted in blue. Click any name to jump straight to it; or use F12 s for the filterable popup. The second row is the normal bar (the current session's window tabs + HPC readouts). It's defined as status-format[0] / status-format[1] in ~/.tmux.conf; to drop back to a single line, set -g status on.

To leave tmux entirely: F12 d. Sessions stay alive. Come back with tproj_a, tproj_b, tproj_c, tsandbox, tproj_d, tproj_e, tadmin — or tmux attach -t proj_a directly.

Admin session tadmin — cluster management

tadmin is the cross-cutting session for installing software, editing configs, driving agents through admin work, and watching cluster-wide state. Different layout from the project sessions because the tasks are different.

1:ops daily driver

Full-pane bash. Where you run install commands, conda, pip, downloads. Keep your real work here so the monitoring windows don't get clobbered.

2:agent parallel agents

Two side-by-side panes, just like the project sessions — agents do the heavy lifting of admin work.
· Alt+← / Alt+→ to jump between left/right pane
· Run agent in one, agent (or a second agent) in the other

3:mon monitor

Split vertical:
· left: htop (auto-launched, falls back to top)
· right: watch -n 5 showing disk (df -h ~ /tmp), memory (free -h), load (uptime)

4:edit editor

Bare shell, kept separate so long vim/nvim sessions don't block your ops pane. Launch your editor of choice from here.

5:cluster global view (vertical stripes)

Complete cluster state in four full-width stripes (top to bottom). Sized so all data fits without truncation — the cluster has 100+ compute nodes and 40+ active jobs at any time, which would overflow a typical pane in single-column form.

StripeCommandWhat it shows
1 (top)watch sinfo | colmajor 2Partition summary — state & node lists, 2-column flow
2watch squeue | colmajor 2Every user's jobs (global queue) — 2-column flow, fits 50+ jobs
3gpustat loop (bash fn from ~/.bashrc)GPU compute nodes — type, count, RAM, users; single column (only ~10 nodes)
4 (bottom)watch cpustat | colmajor 3CPU compute nodes — alloc/idle, load, RAM, users; 3-column flow, fits 100+ nodes
Multi-column flow via ~/bin/colmajor. Pipe any line-list through colmajor N to flow it into N column-major columns (down first, then across). Used here so the cluster window can show all 100+ nodes and 40+ jobs without scrolling. Don't substitute pr -T -N — it re-encodes runs of spaces as TAB chars that don't align with column padding, rendering the output garbled.
Difference from project sessions. Project sessions (proj_a, proj_b, …) show your queue (squeue --me) and your build logs. tadmin shows the global queue, all nodes, the GPU map, AND the CPU map for the whole cluster. Don't mix them up.
If the session ever dies (server restart, kill), just run tadmin again — the alias auto-runs ~/bin/tmux-bootstrap to rebuild it from scratch with all four windows wired up. Idempotent and safe to re-run.

A typical "do work" flow walkthrough

  1. tproj_a → you land in window 1 (shell) at ~/project-a.
  2. F12 2 → switch to the agent window — already split into two side-by-side panes, ready for two parallel agents.
    • Alt+← / Alt+→ to jump between the left and right agent pane
    • Run agent in one pane and agent (or a second agent) in the other
    • F12 z to zoom into one pane fullscreen; press again to pop back to the side-by-side view
  3. F12 3 → queue window. watch squeue --me is already running. Glance, leave it.
  4. F12 4 → logs window. Run tail -f Run.log on whatever you care about.
  5. Need a third agent for the same project? F12 2 → F12 - (horizontal split) → run a third agent in the new pane.
  6. Need a second project? F12 d to detach, tproj_b to attach to project-b. Both sessions (and all their agents) keep running concurrently.

Copy mode scrollback + select text

Mouse-wheel scrolls the pane's scrollback. In a shell or TUI app that doesn't capture the mouse (Agent Code, Agent, plain bash), wheel-up enters copy-mode and scrolls back; wheel-down scrolls forward. Mouse-aware apps (vim, less, watch, htop) get the wheel events forwarded and scroll their own buffers. Press q or Esc to exit copy-mode when you're done browsing the scrollback. The red COPY-MODE tag in the status bar makes it obvious when you're in copy-mode.
You'll know when you're in copy-mode: a red COPY-MODE tag appears in the top-left status bar next to the session name. The mode-line on the active pane turns bold yellow.

TUI apps that emit OSC 52 (an agent, Helix, modern Neovim) copy straight to your system clipboard. In an agent, pressing y on a content block sends an OSC 52 escape sequence; tmux's set-clipboard on + allow-passthrough on + terminal-features ",*:clipboard" propagate it through to FastX, which updates your laptop's clipboard. Paste anywhere normally.

Required FastX client-side setting (one-time toggle): in the FastX session settings, enable "Applications in terminal may access clipboard". Without it, FastX silently discards the OSC 52 sequence and your y press has no visible effect outside the TUI.

Cluster shutdown → respawn disaster recovery

Your whole tmux world is recoverable from disk. The running server dies with the node (its socket lives in node-local /tmp), but every artifact needed to rebuild it — ~/.tmux.conf, ~/bin/tmux-bootstrap, the resurrect save, and all your working dirs — lives on a shared parallel filesystem (shared storage), which survives the shutdown. Two rebuild paths below: bootstrap (clean, live dashboards) or resurrect restore (exact cwds + scrollback).
Before you power down: take a fresh save. Continuum auto-save is intentionally OFF, so the snapshot only reflects your last manual save. Capture the latest layout with F12 C-s (or tmux run-shell '~/.tmux/plugins/tmux-resurrect/scripts/save.sh'). The save writes ~/.tmux/resurrect/last + pane_contents.tar.gz (layout + scrollback for every pane).

When the cluster is back and you've opened a new FastX session:

  1. Confirm DISPLAY auto-healed. Open one fresh pane and run echo $DISPLAY (expect :<n>, not hpc-login:111) and xset q. Every interactive shell sources ~/bin/fastx-display.sh, so new panes auto-point at the live FastX session and GUI launches (e.g. an X11 application) work.
  2. Rebuild — pick ONE:
    • Bootstrap path: run tmux-bootstrap (or the tadmin alias) to recreate the project + admin sessions from scratch, then attach. Clean skeleton — running programs are not restored.
    • Resurrect path: if you saved a layout with F12 C-s, restore it with F12 C-r (tmux-resurrect) to bring back the session/window/pane layout and working dirs.
  3. Working dirs (Path A only): cd the 5 shell panes back to their case dirs — listed in your saved snapshot/runbook.
  4. Resume agents: in each project dir, agent --continue (most recent) or agent --resume (pick). Transcripts persist on disk independent of tmux — nothing is lost.
  5. Stale-DISPLAY panes (pre-existing or Path-B restored): source ~/bin/fastx-display.sh in that pane. Optional: tmux setenv -gu DISPLAY; tmux setenv -gu XAUTHORITY.
⚠ Do not whitelist sbatch in @resurrect-processes. The save can contain an sbatch --chdir=… pane; whitelisting it would auto-resubmit a Slurm job on every restore. Leave it to come back as an idle shell.

Why F12 instead of the default Ctrl+B? FastX/gnome-terminal in this setup eats every Ctrl-modifier combination before it reaches tmux. We proved it: Shift+→ worked (no prefix needed) but every Ctrl+X binding failed. F12 is a pure single keystroke with no modifier — nothing upstream intercepts it.
